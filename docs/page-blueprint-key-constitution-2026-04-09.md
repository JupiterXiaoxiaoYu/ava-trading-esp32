# AVE 页面改造蓝图 + 键位宪法 + 能力落位表（2026-04-09）

> 本文建立在 `docs/product-review-table-2026-04-09.md` 之上。
>
> 目标：把产品审查结论进一步落成可执行的页面蓝图、按键规则和能力落位方案，作为后续设计、实现、验收的统一产品契约。

---

## 1. 总体原则

| 原则 | 说明 |
|---|---|
| 少页面 | 默认不扩顶层页面；能用变体/overlay/解释层解决的，不开新页 |
| 强状态机 | 每个页面只承载一个主任务，避免页面同时承担浏览 + 配置 + 编辑 + 提交 |
| 强身份 | 所有与资产相关的关键动作都必须强化资产身份，不允许用户靠 symbol 猜 |
| 强确定性 | 页面跳转、按键、语音意图必须与当前 screen/context 保持一一对应 |
| 强解释 | 超时、失败、延迟结果、仅浏览模式、`N/A` 都必须有产品解释 |
| 低误触 | 高风险动作不得和低风险浏览动作混在同一层随意切换 |
| 设备优先 | 以 320x240、八键、语音助手心智为前提，而不是桌面研究终端心智 |

## 3. 页面改造蓝图（逐页）

### 3.1 `FEED` 蓝图

| 项目 | 当前 | 目标蓝图 | 产品动作 |
|---|---|---|---|
| 页面角色 | 首页、榜单浏览、来源切换 | 保持首页单一角色 | 不扩成研究中心 |
| 顶栏 | 来源/模式标识已存在 | 保持，并强化“当前来源子集”概念 | 将 `Sources` 文案改成“当前支持来源” |
| 列表项 | symbol/price/change | 增加资产身份强化 | 加入 chain 缩写、必要时合约尾号 |
| Search 入口 | Explore 内文案 `FN 说币名` | 补首次闭环 | Search guide 增加示例词、最近一次查询回显 |
| Orders 入口 | Explore 可进入 browse-only 列表 | 增加解释层 | 入口与页头都明确“当前仅查看” |
| 来源切换 | `X` 切来源 | 保留但不扩语义 | 不再给 `X` 增加第三语义 |
| 语音指代 | 可信 selection 已 fail-closed | 保持 | 不允许用旧 cursor/旧 current_token 补推断 |

### 3.2 `SPOTLIGHT` 蓝图

| 项目 | 当前 | 目标蓝图 | 产品动作 |
|---|---|---|---|
| 页面角色 | 单币详情 + 买/卖入口 | 保持 | 不扩成复杂编辑页 |
| 核心信息 | price/chart/risk/holders/liquidity | 强化资产身份 | 顶区加入 chain、contract tail、必要时 source/issuer tag |
| 数据增强 | 暂无 pair/tx detail | 增加“解释而非堆砌” | 以 1-2 行摘要方式接入 pair / tx detail |
| 买入入口 | `A` | 保持 | 仍是主正向动作 |
| 卖出入口 | `X` 快捷卖出 | 保留但加护栏 | 首次卖出触发更强确认文案/红色语义提示 |
| 左右切币 | 依赖 `feed_token_list + feed_cursor` | 保持 | 只做页内浏览，不跨心智模型 |

### 3.3 `CONFIRM` / `LIMIT_CONFIRM` 蓝图

| 项目 | 当前 | 目标蓝图 | 产品动作 |
|---|---|---|---|
| 页面角色 | 高风险交易确认 | 保持 | 继续避免自由编辑 |
| 关键问题 | 超时/回退解释不够 | 增强解释层 | timeout、watchdog、submit-only、deferred 都要有产品文案 |
| 资产信息 | 仅 symbol 为主 | 增强资产身份 | 同步显示 chain、contract tail 或来源标签 |
| 行为 | `A` confirm、`B` cancel、`Y` cancel->portfolio | 保持 | 不增加其他按键语义 |
| 成功后路径 | 直 RESULT 或 notify/feed 等待终态 | 保持技术路径 | 但对用户必须解释当前处于“已提交/等待链上确认” |

### 3.4 `RESULT` 蓝图

| 项目 | 当前 | 目标蓝图 | 产品动作 |
|---|---|---|---|
| 页面角色 | 成功/失败终态展示 | 保持 | 继续手动停留 |
| 当前问题 | 看到了结果，但下一步弱 | 增加 next step | 成功：看持仓/回首页；失败：返回重试/查看原因 |
| 字段表达 | 支持 title / out_amount / amount_usd / tx_id / error | 保持 | 但文案要更产品化 |
| 返回机制 | 任意非 `Y` 键 back；`Y` 去持仓 | 保持 | 不引入自动跳转 |

### 3.5 `PORTFOLIO` 蓝图

| 项目 | 当前 | 目标蓝图 | 产品动作 |
|---|---|---|---|
| 页面角色 | 钱包资产总览、watch/sell | 保持 | 作为二号主页面 |
| 当前问题 | `N/A` 多、钱包来源弱、解释不足 | 增加解释层 | 加 walletinfo / wallet tokens / pnl explanation |
| 列表项 | symbol/value/P&L | 增强资产身份 | 加 chain / 必要时地址尾号 |
| 卖出入口 | `X` 卖出持仓 | 保持但需护栏 | 与 SPOTLIGHT 保持统一风险表达 |
| watch 入口 | `A` 看详情 | 保持 | 不加入二级复杂操作 |

### 3.6 `NOTIFY` 蓝图

| 项目 | 当前 | 目标蓝图 | 产品动作 |
|---|---|---|---|
| 页面角色 | 解释层、提示层、事件层 | 保持 | 不扩成可操作中心 |
| 当前问题 | 任意键消费会吞操作；文案有时偏工程态 | 增强文案 | 强化“发生了什么 / 现在处于什么状态 / 下一步怎么做” |
| 消费键行为 | 任意键关闭并消费 | 暂保留 | 但重要场景尽量把结果导向 `RESULT` 而非只停留在 notify |

### 3.7 条件新增页：`DISAMBIGUATION`

| 项目 | 蓝图 |
|---|---|
| 目标 | 当 symbol 重名、搜索结果含多个高度相似资产、或进入交易前身份不够清晰时，让用户明确确认“到底是哪个资产” |
| 触发条件 | 1) 同 symbol 多资产；2) 搜索结果同名项竞争；3) 下单前身份置信不足 |
| 页面内容 | symbol、chain、合约尾号、来源/发行方短标签、必要时价格/流动性简述 |
| 输入规则 | `UP/DOWN` 选项移动，`A/RIGHT` 进入，`B/LEFT` 返回，`Y` 全局去持仓，`X` 禁用 |
| 页面性质 | 过渡确认页，不承担分析、不承担下单、不承担配置 |
| 新增约束 | 只允许作为 `FEED/SEARCH -> SPOTLIGHT` 或 `SPOTLIGHT -> CONFIRM` 的中间层，不可成为常驻导航页 |

## 4. 键位宪法（正式版）

### 4.1 宪法总则

| 条款 | 内容 |
|---|---|
| 第一条 | 键位语义必须服务于“短路径、低误触、强确定性” |
| 第二条 | 系统保留键不得因页面便利被临时复用 |
| 第三条 | 高风险动作键不得持续膨胀其语义范围 |
| 第四条 | 页面可复用的键必须保持同一语义家族 |
| 第五条 | 任何新增页面/能力必须先声明键位占用，禁止后补 |

### 4.2 保留键

| 键位 | 宪法定义 | 约束 |
|---|---|---|
| `Y` | 全局去 `PORTFOLIO` | 永不复用 |
| `FN/PTT/F1` | 语音监听控制键 | 永不复用 |
| `P` | 仅模拟器 mock 控制 | 不进入真机产品契约 |

### 4.3 可复用键

| 键位 | 允许的语义家族 | 不允许的语义 |
|---|---|---|
| `A` | 主进入、主确认、主执行 | 二级工具、危险快捷卖出、模式切换 |
| `B` | 返回、取消、关闭、退出局部层 | 成交、确认、全局模式切换 |
| `UP/DOWN` | 线性选择、列表移动、时间周期切换 | destructive action |
| `LEFT/RIGHT` | 页内切换、进入下一级、局部刷新 | 全局跳转、交易确认 |

### 4.4 高风险键特别法：`X`

| 条目 | 宪法结论 |
|---|---|
| 现状 | `FEED=source switch`；`SPOTLIGHT/PORTFOLIO=sell` |
| 结论 | 暂时接受，但必须冻结，不得新增第三语义 |
| 风险防护 | 资产上下文页必须显式显示 `[X] SELL`；FEED 必须显式显示 `X SOURCE` |
| 禁用范围 | `CONFIRM`、`LIMIT_CONFIRM`、`RESULT`、`NOTIFY`、`DISAMBIGUATION` 不给 `X` 新语义 |
| 后续审查门槛 | 若未来想改 `X`，必须先重做整套输入矩阵评审 |

### 4.5 新页面键位预留表

| 页面 | `UP/DOWN` | `LEFT/RIGHT` | `A` | `B` | `X` | `Y` |
|---|---|---|---|---|---|---|
| `DISAMBIGUATION` | 选项移动 | 左退右进 | 选中进入 | 返回上一步 | 禁用 | 全局持仓 |
| Wallet explanation overlay | 可选滚动 | 关闭/无 | 确认关闭或进入详情 | 返回/关闭 | 禁用 | 全局持仓 |
| Order explanation overlay | 可选滚动 | 关闭/无 | 确认关闭或进入挂单详情 | 返回/关闭 | 禁用 | 全局持仓 |

## 5. 能力落位表（产品视角）

### 5.1 必做能力落位

| 能力 | 是否进入现有页 | 落位建议 | 原因 |
|---|---|---|---|
| 资产身份强化 | 是 | `FEED`、`SPOTLIGHT`、`CONFIRM`、`PORTFOLIO` | 贯穿“看谁/买谁/卖谁” |
| 同名币消歧 | 条件新增页 | `DISAMBIGUATION` | 需要强制确认身份 |
| 搜索闭环 | 是 | `FEED_SEARCH` + Search guide | 本质仍属于首页浏览流程 |
| 搜索返回恢复 | 是 | `FEED_SEARCH` | 不值得新开页 |
| 确认超时解释 | 是 | `CONFIRM` / `LIMIT_CONFIRM` / `NOTIFY` / `RESULT` | 属于交易解释层 |
| submit-only / deferred 解释 | 是 | `NOTIFY` + `RESULT` | 属于交易结果体系 |
| Wallet explanation | 是 | `PORTFOLIO` | 属于资产总览解释层 |
| Order explanation | 是 | `FEED_ORDERS` / `RESULT` / `NOTIFY` | 属于订单模式解释层 |

### 5.2 建议集成能力落位

| 能力 | 是否集成 | 落位建议 | 说明 |
|---|---|---|---|
| `walletinfo` | 建议 | `PORTFOLIO` 顶部说明区/解释层 | 解释钱包身份 |
| `walletinfo/tokens` | 建议 | `PORTFOLIO` 数据来源增强 | 解释持仓来源 |
| `address/pnl` | 建议 | `PORTFOLIO` 汇总区 | 解决 `N/A` 心智问题 |
| `address/tx` | 可做 | `PORTFOLIO` 二级解释 | 不应独立成页 |
| `pairs/{pair}` | 建议 | `SPOTLIGHT` 增强摘要区 | 强化交易池语义 |
| `tx/detail` | 建议 | `SPOTLIGHT` / `RESULT` | 强化结果解释 |

### 5.3 明确不落位能力

| 能力 | 不落位原因 |
|---|---|
| approve / transfer | 超出交易助手边界 |
| wallet lifecycle | 超出当前钱包心智模型 |
| chain wallet / self-custody | 风险模型不同 |
| smart wallet / signals / whale / holders 全量页 | 会把产品拉向研究终端 |
| `tx/multi_tx/liq` 主表面实时流 | 与 anti-抢屏原则冲突 |

## 6. 关键流程蓝图

### 6.1 搜索闭环目标流程

| 步骤 | 目标状态 |
|---|---|
| 标准 FEED `B -> Search` | 给出搜索说明 + 示例词 |
| 语音/FN 输入 symbol | 进入 `FEED_SEARCH` |
| 命中唯一资产 | 直接进 `SPOTLIGHT` |
| 命中多同名资产 | 进入 `DISAMBIGUATION` |
| 详情后返回 | 回 `FEED_SEARCH`，保留 query + cursor |

### 6.2 交易解释目标流程

| 场景 | 目标表达 |
|---|---|
| 用户确认后已提交但未终态 | 明确告诉用户“订单已提交，正在等待链上确认” |
| 倒计时超时 | 明确告诉用户“确认超时，系统已自动取消，本次未成交” |
| watchdog 超时 | 明确告诉用户“等待确认超时，稍后可在订单/结果中查看” |
| deferred result | 明确告诉用户“当前有更高优先级确认流程，结果将在稍后显示” |
| 失败 | 给出原因 + 下一步建议 |

### 6.3 Wallet / Order 解释层目标流程

| 表面 | 目标表达 |
|---|---|
| `PORTFOLIO` | 解释 wallet 来源、余额来源、为什么 P&L 可能是 `N/A` |
| `FEED_ORDERS` | 解释“当前仅查看”，以及管理动作的边界 |
| `RESULT` | 若是撤单结果，明确说明是“订单状态变化”，不是资产成交 |

## 7. 产品蓝图最终判断

| 问题 | 结论 |
|---|---|
| 未来是否需要很多新页面 | 不需要 |
| 最小可接受新增页是什么 | `DISAMBIGUATION` |
| 未来的能力承载重心在哪 | `FEED` / `SPOTLIGHT` / `PORTFOLIO` / `RESULT` 的解释增强 |
| 键位治理是否需要单独成文 | 需要，且必须先于实现 |
| 这份蓝图的用途 | 作为后续任务拆解与实现验收的产品基准 |
